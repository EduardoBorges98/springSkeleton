01

SkeletonApplication.java

Ponto de entrada

É o arquivo principal do projeto — o ponto de entrada da aplicação. Quando você executa mvn spring-boot:run, o Java chama o método main() desta classe. O Spring Boot então varre todo o pacote em busca de componentes para registrar no contexto.

src/main/java/com/skeleton/SkeletonApplication.java 
@SpringBootApplication   // = @Configuration + @EnableAutoConfiguration + @ComponentScan
@EnableCaching            // ativa @Cacheable, @CachePut, @CacheEvict nos services
@EnableAsync              // ativa @Async para métodos que rodam em threads separadas
@EnableScheduling         // ativa @Scheduled para tarefas periódicas (cron jobs)
public class SkeletonApplication {
    public static void main(String[] args) {
        SpringApplication.run(SkeletonApplication.class, args);
    }
}

O que cada anotação faz

@SpringBootApplication
Atalho para três anotações: @Configuration (essa classe é fonte de beans), @EnableAutoConfiguration (Spring detecta e configura automaticamente o que foi adicionado — ex: se tem PostgreSQL no classpath, configura datasource), e @ComponentScan (varre o pacote com.skeleton inteiro buscando @Component, @Service, @Repository, @Controller).
@EnableCaching
Sem isso as anotações @Cacheable / @CachePut / @CacheEvict no UserServiceImpl são ignoradas silenciosamente. Esta anotação diz ao Spring para criar um proxy em volta dos métodos anotados e interceptar as chamadas para gerenciar o cache.
@EnableAsync
Permite usar @Async em qualquer método, fazendo ele executar em uma thread separada sem bloquear a chamada original. Útil para envio de emails, notificações, etc.
@EnableScheduling
Ativa o agendador de tarefas do Spring. Com isso você pode usar @Scheduled(cron = "0 0 2 * * *") em qualquer bean para rodar tarefas periodicamente.

Segurança

É o coração da configuração de segurança. Define quem pode acessar o quê, configura CORS, desativa CSRF (desnecessário em APIs stateless), e conecta o filtro JWT à cadeia de filtros do Spring.

src/main/java/com/skeleton/config/SecurityConfig.java 
@Configuration
@EnableWebSecurity         // ativa o módulo de segurança web do Spring
@EnableMethodSecurity      // ativa @PreAuthorize nos controllers
@RequiredArgsConstructor   // Lombok: gera construtor com os campos final
public class SecurityConfig {

    private final JwtAuthenticationFilter jwtAuthFilter;
    private final UserDetailsService userDetailsService;

    @Value("${app.cors.allowed-origins}")
    private String allowedOrigins; // lido do .env via application.yml

    // Rotas que não precisam de token JWT
    private static final String[] PUBLIC_URLS = {
        "/auth/**",          // login e registro
        "/actuator/health",   // health check para kubernetes/ECS
        "/v3/api-docs/**",    // spec OpenAPI (JSON)
        "/swagger-ui/**",     // interface Swagger
        "/swagger-ui.html"
    };

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(AbstractHttpConfigurer::disable)          // CSRF não faz sentido em JWT stateless
            .cors(cors -> cors.configurationSource(corsConfigurationSource()))
            .sessionManagement(s -> s
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS)) // zero sessão HTTP
            .authorizeHttpRequests(auth -> auth
                .requestMatchers(PUBLIC_URLS).permitAll()   // rotas públicas
                .requestMatchers(HttpMethod.OPTIONS, "/**").permitAll() // preflight CORS
                .anyRequest().authenticated()               // todo o resto exige JWT
            )
            .authenticationProvider(authenticationProvider())
            .addFilterBefore(jwtAuthFilter,
                UsernamePasswordAuthenticationFilter.class) // JWT roda antes do filtro padrão
            .build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder(); // fator 12 por padrão
    }
}

Anotações e beans explicados

@EnableMethodSecurity
Sem isso o @PreAuthorize("hasRole('ADMIN')") nos controllers é ignorado. Esta anotação instrui o Spring a criar proxies em volta dos métodos anotados para verificar permissões antes de executar.
STATELESS
O Spring Security não criará nem usará sessões HTTP (HttpSession). Cada requisição é autenticada do zero via JWT. Isso é essencial para APIs horizontalmente escaláveis — qualquer instância valida qualquer token.
addFilterBefore
Posiciona o JwtAuthenticationFilter antes do UsernamePasswordAuthenticationFilter na cadeia. Assim o usuário já está autenticado no SecurityContext quando os filtros posteriores rodam.
DaoAuthenticationProvider
Conecta o UserDetailsService (que carrega o usuário do banco) ao PasswordEncoder (BCrypt). É este provider que o AuthenticationManager usa no login para verificar as credenciais.
Por que CSRF está desabilitado?

CSRF (Cross-Site Request Forgery) só é relevante quando o servidor usa cookies de sessão. Como usamos JWT no header Authorization, o browser nunca envia o token automaticamente para outros sites — o ataque CSRF não é possível.

JpaConfig.java

Dados

Configura a auditoria automática do JPA. Graças a esta classe, campos como createdBy e updatedBy na BaseEntity são preenchidos automaticamente com o email do usuário logado — sem nenhuma linha de código manual no service.

src/main/java/com/skeleton/config/JpaConfig.java 
@Configuration
@EnableJpaAuditing(auditorAwareRef = "auditorProvider")
// ^^ diz ao Spring: "quando precisar saber quem é o auditor atual,
//    usa o bean chamado auditorProvider"
public class JpaConfig {

    @Bean
    public AuditorAware<String> auditorProvider() {
        // AuditorAware é uma interface com um único método: getCurrentAuditor()
        // Implementamos como lambda: sempre retorna o usuário autenticado
        return () -> {
            Authentication auth = SecurityContextHolder.getContext().getAuthentication();
            if (auth == null || !auth.isAuthenticated()) {
                return Optional.of("system"); // chamadas sem autenticação (ex: migrations)
            }
            return Optional.of(auth.getName()); // retorna o email do usuário logado
        };
    }
}
Como o fluxo funciona na prática

Quando você chama userRepository.save(user), o Spring detecta que é um novo registro e chama auditorProvider.getCurrentAuditor() internamente. O resultado é colocado no campo createdBy. No próximo save(), o mesmo valor vai para updatedBy. Você nunca precisa setar esses campos manualmente.

CacheConfig.java

Performance

Configura o Redis como backend de cache. Define TTL padrão de 30 minutos, serialização em JSON e expõe dois beans: o RedisCacheManager (usado pelas anotações @Cacheable) e o RedisTemplate (para operações manuais no Redis).

src/main/java/com/skeleton/config/CacheConfig.java 
@Configuration
public class CacheConfig {

    @Bean
    public RedisCacheManager cacheManager(RedisConnectionFactory factory) {
        // ObjectMapper configurado para incluir tipo no JSON
        // (necessário para desserializar corretamente objetos genéricos)
        ObjectMapper om = new ObjectMapper();
        om.activateDefaultTyping(LaissezFaireSubTypeValidator.instance,
            ObjectMapper.DefaultTyping.NON_FINAL, JsonTypeInfo.As.PROPERTY);

        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(30))  // expira em 30 minutos
            .disableCachingNullValues()         // não salva null no cache
            .serializeKeysWith(...)             // chave como String simples
            .serializeValuesWith(...);          // valor como JSON

        return RedisCacheManager.builder(factory)
            .cacheDefaults(config).build();
    }

    @Bean
    public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
        // Bean para operações manuais: redisTemplate.opsForValue().set("key", val)
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        template.setKeySerializer(new StringRedisSerializer());
        template.setValueSerializer(new GenericJackson2JsonRedisSerializer());
        return template;
    }
}
TTL 30 min
Após 30 minutos o Redis descarta a entrada. A próxima chamada vai ao banco e reabastece o cache.
disableCachingNullValues
Evita salvar null no cache, o que poderia esconder dados que ainda não existem mas serão criados depois.
DefaultTyping
O Jackson inclui o nome da classe no JSON salvo no Redis. Isso permite desserializar corretamente ao ler.
RedisTemplate
Use quando precisar de operações Redis além do cache: listas, sets, pub/sub, expiração customizada, etc.

BaseEntity.java

Entity

Classe abstrata que toda entidade do projeto herda. Ela fornece ID em UUID, campos de auditoria automática e suporte a soft delete. Usar @MappedSuperclass faz com que os campos sejam mapeados na tabela de cada entidade filha, sem criar uma tabela separada para BaseEntity.

src/main/java/com/skeleton/entity/BaseEntity.java 
@Getter @Setter
@MappedSuperclass                          // campos mapeados nas tabelas filhas
@EntityListeners(AuditingEntityListener.class) // ativa o listener de auditoria do JPA
public abstract class BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.UUID) // UUID gerado pelo banco
    @Column(updatable = false, nullable = false)    // imutável após criação
    private UUID id;

    @CreatedDate
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt; // preenchido automaticamente no INSERT

    @LastModifiedDate
    @Column(name = "updated_at")
    private LocalDateTime updatedAt; // atualizado automaticamente no UPDATE

    @CreatedBy
    @Column(name = "created_by", updatable = false)
    private String createdBy;        // preenchido pelo AuditorAware (JpaConfig)

    @LastModifiedBy
    @Column(name = "updated_by")
    private String updatedBy;        // atualizado pelo AuditorAware

    @Column(name = "deleted_at")
    private LocalDateTime deletedAt; // null = ativo | data = deletado (soft delete)

    public boolean isDeleted() { return deletedAt != null; }

    public void softDelete() {
        this.deletedAt = LocalDateTime.now(); // marca como deletado SEM remover do banco
    }
}

Por que UUID e não Long?

Long sequencial
IDs 1, 2, 3... expõem o tamanho do banco, permitem que atacantes adivinhem outros IDs, e criam problemas em bancos distribuídos (qual instância gera o próximo número?).
UUID aleatório
Completamente imprevisível, seguro para expor em URLs públicas, gerado independentemente por qualquer instância sem coordenação. O único custo é ser maior (16 bytes vs 8 bytes).
Soft delete
Nunca usamos DELETE SQL. Em vez disso, setamos deleted_at = NOW(). Isso preserva histórico, facilita auditoria, permite desfazer deleções e evita erros irreversíveis em produção. Queries filtram com WHERE deleted_at IS NULL.

User.java

Entity

A entidade de usuário do sistema. Estende BaseEntity (recebe UUID, auditoria e soft delete de graça) e adiciona os campos específicos do domínio. O relacionamento com roles usa @ElementCollection — uma tabela separada user_roles — em vez de uma entidade Role própria, pois roles são dados simples que não precisam de ID próprio.

src/main/java/com/skeleton/entity/User.java 
@Entity                    // marca esta classe como tabela do banco
@Table(name = "users")    // nome explícito (evita conflito com palavra reservada "user")
@Getter @Setter
@NoArgsConstructor         // construtor vazio: exigido pelo JPA
@AllArgsConstructor        // construtor com todos os campos
@Builder                   // padrão Builder: User.builder().email("...").build()
public class User extends BaseEntity {

    @Column(nullable = false, unique = true)
    private String email;

    @Column(nullable = false)
    private String password;  // SEMPRE hash BCrypt, nunca texto puro

    @Column(nullable = false)
    private String name;

    @Column(name = "phone_number")
    private String phoneNumber;  // nullable: campo opcional

    @Column(name = "is_active", nullable = false)
    @Builder.Default           // valor padrão quando se usa o Builder
    private boolean active = true;

    @ElementCollection(fetch = FetchType.EAGER)  // carrega junto com o User
    @CollectionTable(name = "user_roles",        // tabela separada
        joinColumns = @JoinColumn(name = "user_id"))
    @Enumerated(EnumType.STRING)                  // salva "ROLE_ADMIN", não 0, 1, 2
    @Column(name = "role")
    private Set<Role> roles;
}
@Builder.Default
Sem isso, ao usar User.builder().email("a").build(), o campo active seria false (valor padrão de boolean em Java). Com @Builder.Default, o Lombok usa o valor declarado (true) como padrão no Builder.
EAGER fetch
As roles são carregadas junto com o usuário em toda consulta. Faz sentido aqui porque o UserDetailsService sempre precisa das roles para criar o Authentication. Se fossem dados pesados, LAZY seria melhor.
EnumType.STRING
Salva o nome da role como texto ("ROLE_ADMIN"). A alternativa ORDINAL salva o índice (0, 1, 2) — perigoso porque adicionar/remover um valor do enum muda o índice de todos os outros.

Role.java (enum)

Enum

Define os papéis de usuário do sistema. Usar enum garante que os valores são finitos e conhecidos em tempo de compilação — impossível criar uma role inválida. O prefixo ROLE_ é uma convenção obrigatória do Spring Security: o método hasRole('ADMIN') internamente busca por ROLE_ADMIN.

src/main/java/com/skeleton/enums/Role.java 
@Getter
@RequiredArgsConstructor  // Lombok: gera construtor com 'description'
public enum Role {
    ROLE_ADMIN("Administrador"),
    ROLE_USER("Usuário"),
    ROLE_MANAGER("Gerente");

    private final String description; // campo extra para exibição na UI
}

// Como o Spring Security usa:
// @PreAuthorize("hasRole('ADMIN')")   → busca ROLE_ADMIN nas authorities
// @PreAuthorize("hasAnyRole('ADMIN', 'MANAGER')")  → qualquer um dos dois
// @PreAuthorize("hasAuthority('ROLE_ADMIN')")  → equivalente, mas explícito

Repository

Interface que estende JpaRepository. O Spring Data JPA gera automaticamente a implementação em runtime — você não precisa escrever código SQL para operações básicas. Os métodos find... e exists... com nomes descritivos também são gerados automaticamente (é o query derivation). As queries com @Query usam JPQL — similar a SQL mas com nomes de classe/campo Java.

src/main/java/com/skeleton/repository/UserRepository.java 
@Repository
public interface UserRepository extends JpaRepository<User, UUID> {
    // JpaRepository já fornece: save(), findById(), findAll(),
    // deleteById(), count(), existsById() e muito mais

    // Query derivation: Spring gera SELECT * FROM users WHERE email = ?
    Optional<User> findByEmail(String email);

    // Query derivation: SELECT count(*) > 0 FROM users WHERE email = ?
    boolean existsByEmail(String email);

    // JPQL: usa nome da CLASSE (User) e do CAMPO (deletedAt), não a tabela/coluna
    @Query("""
        SELECT u FROM User u
        WHERE u.deletedAt IS NULL
        AND (:search IS NULL
             OR LOWER(u.name)  LIKE LOWER(CONCAT('%', :search, '%'))
             OR LOWER(u.email) LIKE LOWER(CONCAT('%', :search, '%')))
        """)
    Page<User> findAllActive(@Param("search") String search, Pageable pageable);
    // Pageable controla: página, tamanho e ordenação
    // Page retorna: conteúdo + metadados (total, páginas, etc.)

    @Query("SELECT u FROM User u WHERE u.email = :email AND u.deletedAt IS NULL")
    Optional<User> findActiveByEmail(@Param("email") String email);
    // Versão com soft delete: ignora usuários deletados
}
JPQL vs SQL

JPQL opera sobre as classes Java, não sobre as tabelas. u.deletedAt refere-se ao campo Java, que o Hibernate traduz para a coluna deleted_at. Isso mantém o código agnóstico ao banco — você pode trocar PostgreSQL por MySQL sem mudar as queries.

UserMapper.java

Mapper

Interface MapStruct para converter entre Entity e DTOs. Em tempo de compilação, o MapStruct analisa os campos de origem e destino e gera código Java puro para fazer o mapeamento. O resultado é idêntico ao que você escreveria manualmente — sem reflexão, sem overhead em runtime.

src/main/java/com/skeleton/mapper/UserMapper.java 
@Mapper(
    componentModel = "spring",  // MapStruct registra o impl como @Component
    nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE
    // ^^ se o campo source for null, não sobrescreve o campo destino
    //    fundamental para o PATCH: só atualiza o que foi enviado
)
public interface UserMapper {

    // User → UserResponse: copia todos os campos com mesmo nome automaticamente
    UserResponse toResponse(User user);

    // CreateUserRequest → User: ignora campos de auditoria e id (gerados depois)
    @Mapping(target = "id",        ignore = true)
    @Mapping(target = "createdAt", ignore = true)
    @Mapping(target = "updatedAt", ignore = true)
    @Mapping(target = "createdBy", ignore = true)
    @Mapping(target = "deletedAt", ignore = true)
    @Mapping(target = "active",    constant = "true") // novo usuário sempre ativo
    User toEntity(CreateUserRequest request);

    // @MappingTarget: não cria novo objeto, ATUALIZA o existente
    // Perfeito para PATCH: só os campos não-null do request são copiados
    @Mapping(target = "email",    ignore = true) // email não pode ser alterado
    @Mapping(target = "password", ignore = true) // senha tem endpoint próprio
    @Mapping(target = "roles",    ignore = true) // roles têm endpoint próprio
    void updateEntity(@MappingTarget User user, UpdateUserRequest request);
}
O que o MapStruct gera em tempo de compilação

O arquivo UserMapperImpl.java é gerado em target/generated-sources/ e se parece exatamente com código que você escreveria manualmente: response.setId(user.getId()); response.setEmail(user.getEmail()); .... Nenhuma reflexão em runtime — é Java puro compilado.

JwtService.java

Segurança

Responsável por criar e validar tokens JWT. Um JWT tem três partes separadas por ponto: header (algoritmo), payload (dados) e signature (prova de autenticidade). O payload não é criptografado — qualquer um pode decodificá-lo. A segurança vem da assinatura: sem a chave secreta, é impossível criar um token válido.

src/main/java/com/skeleton/security/service/JwtService.java 
@Slf4j
@Service
public class JwtService {

    @Value("${app.security.jwt.secret}")
    private String secret;           // chave Base64 de 256+ bits (do .env)

    @Value("${app.security.jwt.expiration-ms}")
    private long expirationMs;       // 86400000 = 24h em ms

    @Value("${app.security.jwt.refresh-expiration-ms}")
    private long refreshExpirationMs; // 604800000 = 7 dias em ms

    // ── Geração ──
    public String generateToken(UserDetails user) {
        return buildToken(new HashMap<>(), user, expirationMs);
    }

    public String generateRefreshToken(UserDetails user) {
        return buildToken(new HashMap<>(), user, refreshExpirationMs);
    }

    private String buildToken(Map<String, Object> claims, UserDetails user, long exp) {
        return Jwts.builder()
            .claims(claims)                              // dados extras no payload
            .subject(user.getUsername())                 // "sub": email do usuário
            .issuedAt(new Date(System.currentTimeMillis())) // "iat": emitido em
            .expiration(new Date(System.currentTimeMillis() + exp)) // "exp"
            .signWith(getSigningKey())                   // assina com HMAC-SHA256
            .compact();                                  // gera a string final
    }

    // ── Validação ──
    public boolean isTokenValid(String token, UserDetails user) {
        String username = extractUsername(token);
        return username.equals(user.getUsername()) && !isTokenExpired(token);
        // verifica: email bate E token não expirou
    }

    // ── Extração ──
    public String extractUsername(String token) {
        return extractClaim(token, Claims::getSubject); // lê o campo "sub"
    }

    private SecretKey getSigningKey() {
        return Keys.hmacShaKeyFor(Decoders.BASE64.decode(secret));
        // converte a string Base64 do .env numa chave criptográfica
    }
}
Por que dois tokens?

O access token tem vida curta (1 dia) e é enviado em toda requisição. Se vazado, expira rápido. O refresh token tem vida longa (7 dias) e é usado apenas para renovar o access token via POST /auth/refresh. Isso evita que o usuário precise logar novamente toda vez que o access token expira.

JwtAuthenticationFilter.java

Segurança

Filtro que intercepta cada requisição HTTP antes de chegar ao controller. Estende OncePerRequestFilter, garantindo que rode exatamente uma vez por request. Se o token for válido, autentica o usuário no SecurityContext. Se não houver token (rotas públicas), simplesmente passa adiante.

src/main/java/com/skeleton/security/filter/JwtAuthenticationFilter.java 
@Slf4j
@Component
@RequiredArgsConstructor
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(request, response, filterChain) {

        // 1. Tenta ler o header "Authorization: Bearer eyJ..."
        final String authHeader = request.getHeader("Authorization");

        // 2. Sem token → passa adiante sem autenticar (rotas públicas)
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        try {
            // 3. Extrai o token (remove "Bearer " do início)
            final String jwt = authHeader.substring(7);

            // 4. Extrai o email do payload do token
            final String username = jwtService.extractUsername(jwt);

            // 5. Se tem email e ainda não autenticou nesta requisição
            if (username != null &&
                SecurityContextHolder.getContext().getAuthentication() == null) {

                // 6. Carrega usuário do banco (com roles)
                UserDetails userDetails = userDetailsService.loadUserByUsername(username);

                // 7. Valida assinatura e expiração
                if (jwtService.isTokenValid(jwt, userDetails)) {

                    // 8. Cria o objeto de autenticação com as authorities (roles)
                    var authToken = new UsernamePasswordAuthenticationToken(
                        userDetails, null, userDetails.getAuthorities()
                    );
                    authToken.setDetails(new WebAuthenticationDetailsSource()
                        .buildDetails(request));

                    // 9. Registra no SecurityContext — a partir daqui
                    //    SecurityContextHolder.getContext().getAuthentication()
                    //    retorna este usuário em qualquer parte da aplicação
                    SecurityContextHolder.getContext().setAuthentication(authToken);
                }
            }
        } catch (Exception ex) {
            log.warn("JWT authentication failed: {}", ex.getMessage());
            // Não retorna 401 aqui: deixa o Spring Security decidir
            // Se a rota exige autenticação, ele retornará 401/403 depois
        }

        // 10. Continua a cadeia de filtros independentemente
        filterChain.doFilter(request, response);
    }
}

UserDetailsServiceImpl.java

Segurança

Implementação da interface UserDetailsService do Spring Security. É a ponte entre o banco de dados e o sistema de autenticação. O Spring chama loadUserByUsername(email) sempre que precisa verificar credenciais — no login e em cada requisição autenticada (via o filtro JWT).

src/main/java/com/skeleton/security/service/UserDetailsServiceImpl.java 
@Service
@RequiredArgsConstructor
public class UserDetailsServiceImpl implements UserDetailsService {

    private final UserRepository userRepository;

    @Override
    @Transactional(readOnly = true)
    public UserDetails loadUserByUsername(String email) throws UsernameNotFoundException {

        // Busca apenas usuários ativos (com soft delete respeitado)
        User user = userRepository.findActiveByEmail(email)
            .orElseThrow(() -> new UsernameNotFoundException("User not found: " + email));

        // Converte os Role enums em GrantedAuthority
        // Spring Security usa GrantedAuthority para @PreAuthorize
        Set<GrantedAuthority> authorities = user.getRoles().stream()
            .map(role -> new SimpleGrantedAuthority(role.name())) // "ROLE_ADMIN"
            .collect(Collectors.toSet());

        // Retorna um UserDetails (objeto do Spring, não nossa entidade User)
        return User.builder()          // User do Spring Security, não do nosso domínio
            .username(user.getEmail())
            .password(user.getPassword())  // hash BCrypt — Spring compara internamente
            .authorities(authorities)
            .disabled(!user.isActive())    // usuário inativo não consegue logar
            .build();
    }
}

src/main/java/com/skeleton/service/impl/UserServiceImpl.java 
@Slf4j
@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)  // default: sem escrita (otimiza queries)
public class UserServiceImpl implements UserService {

    private final UserRepository  userRepository;
    private final UserMapper       userMapper;
    private final PasswordEncoder  passwordEncoder;

    @Override
    @Transactional   // sobrescreve readOnly=true → permite escrita + rollback
    public UserResponse create(CreateUserRequest request) {
        log.info("Creating user: {}", request.getEmail());

        if (userRepository.existsByEmail(request.getEmail()))
            throw new BusinessException("Email already in use: " + request.getEmail());

        User user = userMapper.toEntity(request);                       // DTO → Entity
        user.setPassword(passwordEncoder.encode(request.getPassword())); // hash a senha
        User saved = userRepository.save(user);                         // INSERT
        return userMapper.toResponse(saved);                             // Entity → DTO
    }

    @Override
    @Cacheable(value = "users", key = "#id")
    // 1ª chamada: vai ao banco e salva no Redis com TTL 30min
    // 2ª chamada com mesmo id: retorna do Redis sem tocar no banco
    public UserResponse findById(UUID id) {
        return userMapper.toResponse(findUserById(id));
    }

    @Override
    @Transactional
    @CachePut(value = "users", key = "#id")  // atualiza o cache após salvar
    public UserResponse update(UUID id, UpdateUserRequest request) {
        User user = findUserById(id);
        userMapper.updateEntity(user, request); // só atualiza campos não-null
        return userMapper.toResponse(userRepository.save(user));
    }

    @Override
    @Transactional
    @CacheEvict(value = "users", key = "#id")  // remove do cache
    public void delete(UUID id) {
        User user = findUserById(id);
        user.softDelete();              // seta deleted_at = NOW()
        userRepository.save(user);      // UPDATE (não DELETE)
    }

    // Método privado de suporte: busca com soft delete
    private User findUserById(UUID id) {
        return userRepository.findById(id)
            .filter(u -> !u.isDeleted())  // ignora usuários soft-deletados
            .orElseThrow(() -> new ResourceNotFoundException("User", id));
    }
}
@Transactional(readOnly)
O Hibernate desativa o dirty checking (varredura de mudanças) e o banco pode usar otimizações de leitura. Use sempre em métodos que só consultam. Sobrescreva com @Transactional simples nos métodos de escrita.
@Cacheable
Spring gera um proxy em volta do método. Na primeira chamada, executa o método e salva o resultado no Redis com a chave composta por value + key (ex: "users::550e8400..."). Nas chamadas seguintes com o mesmo ID, retorna do Redis sem executar o método.
@CachePut
Sempre executa o método E atualiza o cache com o resultado. Usado no update para manter o cache sincronizado com o banco.
@CacheEvict
Remove a entrada do cache após a execução. Usado no delete: após soft-deletar, remove do Redis para que consultas futuras não retornem um usuário que deveria estar "deletado".

AuthService.java

Service

Orquestra o fluxo de autenticação: registro, login e renovação de token. Usa o AuthenticationManager do Spring Security para validar credenciais no login — ele internamente chama o UserDetailsService e o PasswordEncoder.

src/main/java/com/skeleton/service/impl/AuthService.java 
@Slf4j @Service @RequiredArgsConstructor
public class AuthService {

    private final UserRepository       userRepository;
    private final UserMapper            userMapper;
    private final PasswordEncoder       passwordEncoder;
    private final JwtService            jwtService;
    private final AuthenticationManager authenticationManager;
    private final UserDetailsService    userDetailsService;

    @Transactional
    public AuthResponse register(CreateUserRequest request) {
        if (userRepository.existsByEmail(request.getEmail()))
            throw new BusinessException("Email already in use");

        User user = userMapper.toEntity(request);
        user.setPassword(passwordEncoder.encode(request.getPassword()));
        user.setRoles(request.getRoles() != null
            ? request.getRoles() : Set.of(Role.ROLE_USER)); // padrão: ROLE_USER

        User saved = userRepository.save(user);
        return buildAuthResponse(saved); // já retorna os tokens
    }

    public AuthResponse login(LoginRequest request) {
        // authenticationManager.authenticate() faz internamente:
        // 1. loadUserByUsername(email) → busca no banco
        // 2. passwordEncoder.matches(senha, hash) → compara
        // Se falhar: lança BadCredentialsException → GlobalExceptionHandler → 401
        authenticationManager.authenticate(
            new UsernamePasswordAuthenticationToken(
                request.getEmail(), request.getPassword()));

        User user = userRepository.findActiveByEmail(request.getEmail()).orElseThrow();
        return buildAuthResponse(user);
    }

    public AuthResponse refresh(RefreshTokenRequest request) {
        String username = jwtService.extractUsername(request.getRefreshToken());
        UserDetails userDetails = userDetailsService.loadUserByUsername(username);

        if (!jwtService.isTokenValid(request.getRefreshToken(), userDetails))
            throw new BusinessException("Token inválido", HttpStatus.UNAUTHORIZED);

        User user = userRepository.findActiveByEmail(username).orElseThrow();
        return buildAuthResponse(user); // gera novos tokens
    }

    private AuthResponse buildAuthResponse(User user) {
        return AuthResponse.builder()
            .accessToken(jwtService.generateToken(toUserDetails(user)))
            .refreshToken(jwtService.generateRefreshToken(toUserDetails(user)))
            .tokenType("Bearer")
            .expiresIn(expirationMs / 1000)   // em segundos para o cliente
            .user(userMapper.toResponse(user))
            .build();
    }
}

AuthController.java

Controller

Expõe os endpoints públicos de autenticação. Não exige token JWT — está nas rotas públicas do SecurityConfig. O @Valid aciona o Bean Validation antes de o método executar: se o DTO for inválido, o GlobalExceptionHandler captura e retorna 422 automaticamente.

src/main/java/com/skeleton/controller/AuthController.java 
@RestController              // = @Controller + @ResponseBody (resposta em JSON)
@RequestMapping("/auth")    // prefixo para todos os endpoints
@RequiredArgsConstructor
@Tag(name = "Authentication") // agrupa no Swagger UI
public class AuthController {

    private final AuthService authService;

    @PostMapping("/register")
    @Operation(summary = "Registrar novo usuário")
    public ResponseEntity<AuthResponse> register(
            @Valid @RequestBody CreateUserRequest request) {
        // @Valid → valida o DTO antes de entrar no método
        // @RequestBody → desserializa o JSON do body para o objeto
        return ResponseEntity.status(201).body(authService.register(request));
    }

    @PostMapping("/login")
    public ResponseEntity<AuthResponse> login(
            @Valid @RequestBody LoginRequest request) {
        return ResponseEntity.ok(authService.login(request));
    }

    @PostMapping("/refresh")
    public ResponseEntity<AuthResponse> refresh(
            @Valid @RequestBody RefreshTokenRequest request) {
        return ResponseEntity.ok(authService.refresh(request));
    }
}

UserController.java

Controller

CRUD completo de usuários. Todos os endpoints exigem JWT (configurado no SecurityConfig). Criação e deleção exigem adicionalmente a role ADMIN via @PreAuthorize. A listagem suporta paginação e busca via query params.

src/main/java/com/skeleton/controller/UserController.java 
@RestController
@RequestMapping("/v1/users")
@RequiredArgsConstructor
@Tag(name = "Users")
@SecurityRequirement(name = "bearerAuth") // mostra cadeado no Swagger
public class UserController {

    private final UserService userService;

    @PostMapping
    @PreAuthorize("hasRole('ADMIN')")  // 403 se não for ROLE_ADMIN
    public ResponseEntity<ApiResponse<UserResponse>> create(
            @Valid @RequestBody CreateUserRequest request) {
        return ResponseEntity.status(201)
            .body(ApiResponse.ok("User created", userService.create(request)));
    }

    @GetMapping("/{id}")
    public ResponseEntity<ApiResponse<UserResponse>> findById(
            @PathVariable UUID id) { // UUID do segmento da URL
        return ResponseEntity.ok(ApiResponse.ok(userService.findById(id)));
    }

    @GetMapping
    public ResponseEntity<ApiResponse<PageResponse<UserResponse>>> findAll(
            @RequestParam(required = false) String search,     // ?search=ana
            @RequestParam(defaultValue = "0")    int page,
            @RequestParam(defaultValue = "20")   int size,
            @RequestParam(defaultValue = "createdAt") String sortBy,
            @RequestParam(defaultValue = "DESC") String sortDir) {
        Sort sort = Sort.by(Sort.Direction.fromString(sortDir), sortBy);
        return ResponseEntity.ok(
            ApiResponse.ok(userService.findAll(search, PageRequest.of(page, size, sort))));
    }

    @PatchMapping("/{id}")   // PATCH = atualização parcial (vs PUT = substituição total)
    public ResponseEntity<ApiResponse<UserResponse>> update(
            @PathVariable UUID id,
            @Valid @RequestBody UpdateUserRequest request) {
        return ResponseEntity.ok(
            ApiResponse.ok("User updated", userService.update(id, request)));
    }

    @DeleteMapping("/{id}")
    @PreAuthorize("hasRole('ADMIN')")
    public ResponseEntity<Void> delete(@PathVariable UUID id) {
        userService.delete(id);
        return ResponseEntity.noContent().build(); // 204 No Content
    }
}

DTOs de request e response

DTOs

DTOs (Data Transfer Objects) são classes simples que definem o contrato da API. Separam o que o cliente vê do que está no banco. Nunca exponha uma Entity diretamente: ela pode ter campos sensíveis (senha) e acoplaria a API ao schema do banco.

dto/request/CreateUserRequest.java
@Data  // Lombok: @Getter + @Setter + @ToString + @EqualsAndHashCode + @RequiredArgsConstructor
public class CreateUserRequest {

    @NotBlank(message = "Name is required")   // não nulo E não vazio
    @Size(min = 2, max = 100, message = "Name must be 2–100 chars")
    private String name;

    @NotBlank(message = "Email is required")
    @Email(message = "Invalid email format")  // valida formato de email
    private String email;

    @NotBlank(message = "Password is required")
    @Size(min = 8, message = "Minimum 8 characters")
    private String password;

    private String phoneNumber;  // sem validação: opcional

    @NotNull(message = "Roles are required")
    private Set<Role> roles;
}
dto/request/UpdateUserRequest.java
@Data
public class UpdateUserRequest {
    // Todos os campos são opcionais (null = não mudar)

public class UserResponse {

Validações disponíveis (Bean Validation)

@NotNull
Campo não pode ser null. Permite string vazia.
@NotBlank
Campo não pode ser null nem string vazia/espaços. Mais restrito que @NotNull para Strings.
@Email
Valida formato básico de email: precisa ter @ e domínio válido.
@Size
Controla tamanho de String, Collection ou array. min e max são opcionais.
@Min / @Max
Para campos numéricos. @Min(0) garante que um preço não seja negativo.
@Pattern
Valida com regex. @Pattern(regexp = "\\d{10,11}") para telefone, por exemplo.

ApiResponse<T> e PageResponse<T>

DTOs

Envelopes que padronizam todas as respostas da API. ApiResponse<T> garante que o cliente sempre recebe a mesma estrutura, independentemente do endpoint. PageResponse<T> encapsula os dados de paginação do Spring em um formato serializado limpo.

dto/response/ApiResponse.java
@Getter
@Builder
@JsonInclude(JsonInclude.Include.NON_NULL) // omite campos null no JSON
public class ApiResponse<T> {
    private final boolean success;
    private final String message;       // null em respostas sem mensagem
    private final T data;               // o objeto retornado (genérico)
    @Builder.Default
    private final LocalDateTime timestamp = LocalDateTime.now();

    // Factory methods: evitam chamar o builder diretamente nos controllers
    public static <T> ApiResponse<T> ok(T data) { ... }
    public static <T> ApiResponse<T> ok(String message, T data) { ... }
}

Exceções e GlobalExceptionHandler

Exceções

O projeto usa exceções customizadas que carregam o status HTTP apropriado. Um único handler global (@RestControllerAdvice) captura tudo e retorna respostas padronizadas. Sem try/catch espalhado no código — lança a exceção, o handler resolve.

exception/BusinessException.java
@Getter
public class BusinessException extends RuntimeException {
    private final HttpStatus status;

    public BusinessException(String message) {
        super(message);
        this.status = HttpStatus.BAD_REQUEST; // padrão: 400
    }
    public BusinessException(String message, HttpStatus status) {
        super(message);
        this.status = status; // status customizado: 401, 403, 409, etc.
    }
}

// ResourceNotFoundException (404):
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String resource, Object id) {
        super(resource + " not found with id: " + id);
    }
}
exception/GlobalExceptionHandler.java
@Slf4j
@RestControllerAdvice  // intercepta exceções de todos os @RestController
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ErrorResponse> handleBusiness(BusinessException ex, ...) {
        log.warn("Business: {}", ex.getMessage());
        return ResponseEntity.status(ex.getStatus()).body(build(ex.getStatus(), ...));
    }

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(... , ...) {
        return ResponseEntity.status(404).body(build(NOT_FOUND, ex.getMessage(), req));
    }

    // Captura erros de @Valid: retorna 422 com mapa de campos
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(... ex, ...) {
        Map<String, String> fields = new HashMap<>();
        ex.getBindingResult().getAllErrors().forEach(err -> {
            String field = ((FieldError) err).getField();
            fields.put(field, err.getDefaultMessage()); // mensagem da anotação
        });
        ErrorResponse body = build(UNPROCESSABLE_ENTITY, "Validation failed", req);
        body.setFields(fields);
        return ResponseEntity.status(422).body(body);
    }

    // 403, 401, 500...
    @ExceptionHandler(AccessDeniedException.class)  // → 403
    @ExceptionHandler(BadCredentialsException.class) // → 401
    @ExceptionHandler(Exception.class)              // → 500 (fallback)
}

// Formato de resposta de erro (ErrorResponse):
// {
//   "timestamp": "2024-05-18T10:30:00",
//   "status":  422,
//   "error":   "Unprocessable Entity",
//   "message": "Validation failed",
//   "path":    "/api/v1/users",
//   "fields":  { "email": "Invalid email format" }  ← só em erros de validação
// }

Testes unitários e de integração

Testes

O projeto tem dois níveis de teste. Unitários com Mockito testam a lógica de cada classe isoladamente, sem subir o Spring — são rápidos (milissegundos). Integração com Testcontainers sobem um PostgreSQL real em Docker e testam o fluxo completo — são lentos mas testam o que realmente importa.

test/.../service/UserServiceTest.java
@ExtendWith(MockitoExtension.class)  // sem Spring, apenas Mockito
@DisplayName("UserService")
class UserServiceTest {

    @Mock UserRepository  userRepository;  // objeto falso controlado pelo teste
    @Mock UserMapper      userMapper;
    @Mock PasswordEncoder passwordEncoder;

    @InjectMocks                             // Mockito injeta os @Mock acima aqui
    UserServiceImpl userService;

    @Nested
    @DisplayName("create()")
    class Create {

        @Test
        @DisplayName("deve criar usuário com sucesso")
        void shouldCreateUser() {
            // Arrange: configura o que os mocks devem retornar
            when(userRepository.existsByEmail(any())).thenReturn(false);
            when(userMapper.toEntity(any())).thenReturn(user);
            when(userRepository.save(any())).thenReturn(user);
            when(userMapper.toResponse(user)).thenReturn(userResponse);

            // Act: executa o método
            UserResponse result = userService.create(request);

            // Assert: verifica o resultado
            assertThat(result.getEmail()).isEqualTo("john@example.com");
            verify(userRepository).save(any(User.class)); // confirma que salvou
        }

        @Test
        @DisplayName("deve lançar exceção quando email existe")
        void shouldThrowWhenEmailExists() {
            when(userRepository.existsByEmail(any())).thenReturn(true);

            assertThatThrownBy(() -> userService.create(request))
                .isInstanceOf(BusinessException.class)
                .hasMessageContaining("Email already in use");

            verify(userRepository, never()).save(any()); // confirma: NÃO salvou
        }
    }
}
test/.../integration/BaseIntegrationTest.java
@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)
@ActiveProfiles("test")    // usa perfil test do application.yml
@Testcontainers           // ativa gerenciamento de containers Docker
public abstract class BaseIntegrationTest {

    @Container              // sobe 1 container para toda a test class (reusado)
    static PostgreSQLContainer<?> postgres =
        new PostgreSQLContainer<>("postgres:16-alpine")
            .withDatabaseName("skeleton_test");

    @DynamicPropertySource  // injeta URL/user/pass do container no contexto Spring
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url",      postgres::getJdbcUrl);
        registry.add("spring.datasource.username",  postgres::getUsername);
        registry.add("spring.datasource.password",  postgres::getPassword);
        registry.add("spring.flyway.enabled", () -> "true");
    }
    // Todos os testes que estenderem esta classe terão banco PostgreSQL real
}

@Nested
Agrupa testes relacionados dentro de uma classe. Ex: todos os testes de create() ficam juntos, melhorando legibilidade.
@DisplayName
Nome legível que aparece no relatório de testes. Escreva em linguagem natural: "deve lançar exceção quando email existe".
verify()
Verifica se um método mock foi chamado, quantas vezes e com quais argumentos. never() verifica que NÃO foi chamado.
@DynamicPropertySource
Injeta propriedades dinamicamente no contexto Spring após o container Docker subir. Essencial para passar a URL do banco gerada pelo Testcontainers.

Como as variáveis de ambiente funcionam

A sintaxe ${VAR:default} lê a variável de ambiente VAR. Se ela não existir, usa o valor após os dois pontos. Em produção, você define DB_URL, JWT_SECRET etc. nas variáveis do servidor (ECS, Kubernetes, Heroku...). Localmente, crie um arquivo .env e use -Dspring.config.import=optional:file:.env[.properties] ou configure no IDE. O arquivo .env nunca deve ser commitado no git.
